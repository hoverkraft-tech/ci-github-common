## Overview

Opinionated GitHub Actions and reusable workflows for foundational continuous-integration automation that can be shared across Hoverkraft projects.

## Actions

### Workflow & repository automation

- [Checkout](actions/checkout/README.md) ‚Äî event-aware drop-in replacement for `actions/checkout` that supports issue comment triggers.
- [Create and merge pull request](actions/create-and-merge-pull-request/README.md) ‚Äî opens a pull request, rebases, and merges it with the GitHub Actions bot identity.
- [Create or update comment](actions/create-or-update-comment/README.md) ‚Äî adds or updates comments on issues and pull requests idempotently.
- [Get GitHub Actions bot user](actions/get-github-actions-bot-user/README.md) ‚Äî retrieves the profile information for the GitHub Actions bot.
- [Get issue number](actions/get-issue-number/README.md) ‚Äî extracts the relevant issue number from the current workflow context.

### Matrix & workflow data helpers

- [Get matrix outputs](actions/get-matrix-outputs/README.md) ‚Äî aggregates outputs across matrix jobs for downstream steps.
- [Set matrix output](actions/set-matrix-output/README.md) ‚Äî writes structured outputs that can be consumed by other matrix jobs.

### Repository insights & utilities

- [Parse CI reports](actions/parse-ci-reports/README.md) ‚Äî parses CI reports (tests, linting, coverage) into GitHub summaries and Markdown for PR comments.
- [Repository owner is organization](actions/repository-owner-is-organization/README.md) ‚Äî checks whether the repository owner is an organization.
- [Slugify](actions/slugify/README.md) ‚Äî converts free-form strings into GitHub-friendly slugs.

## Reusable Workflows

### Community & issue hygiene

- [Greetings](.github/workflows/greetings.md) ‚Äî welcomes first-time issue reporters and pull request authors.
- [Need fix to Issue](.github/workflows/need-fix-to-issue.md) ‚Äî labels issues that require follow-up fixes.
- [Stale](.github/workflows/stale.md) ‚Äî automatically marks and closes stale issues and pull requests.

### Quality gates

- [Linter](.github/workflows/linter.md) ‚Äî runs Super Linter and shared formatting checks across the repository.
- [Semantic pull request](.github/workflows/semantic-pull-request.md) ‚Äî enforces semantic pull request titles before merging.

### Action structure pattern

All actions follow a consistent structure:


### Development standards

1. **Pinned dependencies** ‚Äì reference external actions by commit SHA (for example `actions/checkout@08c6903c‚Ä¶`).
2. **Composite-first** ‚Äì encapsulate logic in composite actions and keep scripts minimal.
3. **Early validation** ‚Äì validate inputs up front using GitHub Script or shell guards and fail fast with helpful messages.
4. **Consistent branding** ‚Äì set `author: hoverkraft` and `branding.color: blue` with an appropriate icon in every `action.yml`.

## Development Workflow

```bash
make lint        # Run Super Linter (dockerized)
make lint-fix    # Run Super Linter with autofix enabled
gh act -W .github/workflows/<workflow>.yml  # Exercise workflows locally when needed
```

## File Structure

```text
actions/                      # Composite GitHub Actions
.github/workflows/            # Reusable workflows (.yml) with accompanying docs (.md)
Dockerfile                    # Super Linter container definition
Makefile                      # Lint helpers
```
